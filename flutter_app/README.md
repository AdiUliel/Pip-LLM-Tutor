# Pip — Parent App

Flutter + Firebase **parent companion** for Pip, the ESP32-based voice tutor for
kids (ages 7–11). This app does **no tutoring** — that lives on the physical
device. The app lets a parent pair the device, configure the child, upload
homework materials, view reports + mood/accuracy trends, and monitor the device.

Hosted URL: https://llm-tutor-d721e.web.app

---

## Quick start

```bash
flutter pub get
flutter run        # local dev
flutter build web --release && firebase deploy --only hosting   # publish
```

The app talks to the real Firebase project (`llm-tutor-d721e`). Sign-in supports
**Email/Password** and **Google Sign-In** (popup on web, native picker on
Android); parent + child + material + session data live in Firestore; uploads go
to Storage.

Android APK build (JDK 17 required):

```bash
JAVA_HOME=/opt/homebrew/opt/openjdk@17/libexec/openjdk.jdk/Contents/Home \
flutter build apk --release
```

## Reconfigure the Firebase project

If you need to point the app at a different Firebase project:

1. Install the Firebase CLI and the FlutterFire CLI:
   ```bash
   curl -sL https://firebase.tools | bash
   dart pub global activate flutterfire_cli
   ```
2. From this directory, run `flutterfire configure` and pick the project; this
   regenerates `lib/firebase_options.dart`.

## Architecture

```
lib/
  main.dart            Firebase init + MultiProvider + provider wiring + runApp
  app.dart             MaterialApp · RTL (Hebrew) locale · auth-gate routing
  theme.dart           Design tokens
  constants.dart       Every tunable (no magic numbers) + Hebrew labels + enums
  firebase_options.dart  Generated by `flutterfire configure`
  models/              ParentUser · Child · Session · QuestionLog · MaterialDoc · DeviceState
  providers/           AuthProvider · ChildProvider · ConfigProvider · DeviceProvider · StatsProvider
  services/            FirebaseService + DeviceSyncService (abstract interfaces + real
                       implementations) · FcmService (push notifications)
  screens/             login · onboarding intro · setup wizard · shell (bottom-nav) ·
                       dashboard · child config · material upload/detail ·
                       reports hub/reports · session detail · trends ·
                       device monitor · settings · admin view
  widgets/             PairingSheet (device pairing) · RobotFace · charts/cards ·
                       OfflineBanner · wizard/stepper components
  utils/               offline_queue · validators · notifications
```

**Strict rule (per course req):** screens never touch Firestore directly.
Screens → providers → services. Every external IO sits behind a service
interface (`FirebaseService`, `DeviceSyncService`).

## Firestore data model (shared with the ESP32 device)

```
parents/{parentId}                     # WRITTEN BY APP
  name, email, createdAt

children/{childId}                     # WRITTEN BY APP
  parentId, name, age,
  subjectsEnabled: ["math","english"],
  topicFocus, level,
  settings: { sessionMinutes, breakEveryMinutes, dailyLimitMinutes,
              breakEveryQuestions, breakAfterMinutes,      # break policy
              screenOffMinutes, deviceSleepMinutes },      # idle power policy
  deviceId,                            # pairing: set via PairingSheet; a Cloud
  createdAt                            # Function keeps it unique per device

materials/{materialId}                 # WRITTEN BY APP
  childId, subject, title,
  items: [ { question, answer } ],     # extracted/validated Q&A pairs
  fileUrl, status, uploadedAt

deviceState/{deviceId}                 # device writes; app reads
  online, lastHeartbeat,
  status: "idle"|"asking"|"listening"|"feedback"|"break"|"error",
  activeSubject, currentQuestion

sessions/{sessionId}                   # device writes; app reads
  childId, subject, startedAt, endedAt,
  questionsAsked, correctCount, wrongCount, accuracy,
  starsEarned, longestStreak, moodSummary
  questions/ (subcollection)
    { prompt, expectedAnswer, childAnswerTranscript,
      correct, mood, difficulty, askedAt }
```

Pairing: the device shows a `TUTOR-XXXXXX` code on screen; the parent enters it
in the PairingSheet, which links `children/{id}.deviceId` to the device's UID
("newest wins" — a Cloud Function unlinks any other child). The app derives
"online" from the device heartbeat (timeout in `constants.dart`).
